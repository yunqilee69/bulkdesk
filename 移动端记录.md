# BulkDesk 移动端说明

> 本文记录移动端的产品范围与业务约束，用于后续设计和拆分实施任务；当前仅完善说明，不进行客户端、接口或数据库实现。

## 1. 定位与目标

移动端采用**一个具备原生体验的跨端 App**，服务批发业务中的客户采购和商家日常经营。App 按身份提供两个相互隔离的工作台，而不是拆分为两个独立应用：

- **客户工作台**：供游客和已登录客户浏览商品、提交采购需求、查看订单与维护个人资料。
- **商家工作台**：供商家管理员和员工查询商品、处理订单、查看和操作库存、查询客户资料。

这样可减少安装与维护成本，并支持同一人既是客户又是商家员工的场景。不同工作台必须使用独立的首页、底部导航、路由入口和缓存空间，不能把客户与商家功能混排在同一个页面中。

当前按普通批发贸易工具设计，暂不纳入特殊行业监管、支付牌照或行业备案范围；但仍需遵守基本的登录鉴权、权限控制、最小化展示客户信息和操作留痕要求。

## 2. 身份与工作台

| 身份 | 可进入的工作台 | 主要能力 |
| --- | --- | --- |
| 游客 | 客户工作台（有限） | 浏览商品、维护本机购物车、生成临时下单二维码、查询本机保存的临时订单状态 |
| 已登录客户 | 客户工作台 | 查看适用价格、提交订单、查询本人订单、维护个人资料 |
| 商家普通员工 | 商家工作台 | 查询商品、库存、订单和客户；仅执行其后端权限允许的业务操作 |
| 商家管理员 | 商家工作台 | 包含普通员工能力；涉及员工、基础资料等管理能力仍以现有后端权限为准 |

### 2.1 多角色切换

- 一个登录账号如同时关联客户身份和商家员工身份，可在“我的/身份切换”中主动切换工作台。
- 切换为显式操作，不在页面间自动跳转；切换时清理上一工作台的页面栈、筛选条件和敏感缓存，避免误操作或数据串联。
- 仅展示当前身份已授权的入口。隐藏菜单不是权限控制，服务端仍必须对每次写操作鉴权。
- 游客进入客户工作台无需登录；进入商家工作台、查看客户本人订单详情或使用会员价前必须完成登录。

## 3. 信息架构

### 3.1 客户工作台

建议使用四个一级入口：

| 一级入口 | 页面与能力 |
| --- | --- |
| 首页 | 商品搜索、分类浏览、商品详情、库存/可售提示、常用商品与活动位（活动能力后续按实际业务决定） |
| 购物车 | 本机维护商品、数量与备注；校验商品状态和可售数量后进入下单流程 |
| 订单 | 已登录客户查看本人订单；游客查看本机保存的临时订单记录及其处理状态 |
| 我的 | 登录/退出、客户资料、收货信息（后续启用时）、身份切换、隐私与本机数据管理 |

商品展示以现有商品、分类、品牌、图片、规格/单位和销售状态为准。客户最终成交价必须由服务端按客户等级会员价优先、标准售价兜底计算；客户端只能展示价格，不可自行计算或覆盖成交单价。

### 3.2 商家工作台

建议使用四个一级入口：

| 一级入口 | 页面与能力 |
| --- | --- |
| 工作台 | 待处理订单、库存预警、快捷扫码、常用查询入口；具体指标以现有可用数据为准 |
| 订单 | 订单搜索、详情、扫码获取游客临时下单信息、发货、确认付款、完成或取消订单 |
| 库存 | 商品库存查询、按仓库筛选、库存流水查看，以及经授权的入库、出库、调拨、盘点入口 |
| 我的 | 当前员工信息、当前身份、身份切换、退出登录与应用设置 |

商品和客户查询可由工作台或订单/库存页面进入，不单独占用一级 Tab。员工管理、客户等级、仓库和供应商等低频基础资料维护，默认不列入移动端一期；如后续确有现场维护需求，再单独评估权限和表单复杂度。

## 4. 客户下单与游客协助下单

### 4.1 已登录客户下单

1. 客户浏览商品并加入购物车。
2. 客户提交商品、数量、备注及后续需要的收货信息。
3. 服务端校验商品销售状态、客户身份、会员价和可用库存，并创建正式订单。
4. 正式订单创建后进入 `placed` 状态，系统锁定对应库存。
5. 客户可在订单页查询状态；商家按照既有流程处理订单。

移动端不得绕过现有订单状态机。正式订单仅允许遵循：

```text
placed -> shipped -> paid -> completed
```

在 `completed` 前可取消；取消、发货和库存恢复规则以现有后端订单与库存服务为唯一事实来源。

### 4.2 游客协助下单

游客不直接创建正式订单，而是创建一个**临时下单单据**，用于把采购意向交给商家确认：

1. 游客在客户工作台浏览商品并维护本机购物车。
2. 游客提交商品、数量和备注后，服务端创建临时单据并返回短时有效、不可猜测的二维码载荷。
3. App 展示二维码；商家在商家工作台使用扫码功能获取临时单据。
4. 商家核对商品、数量和客户信息，选择或创建正式客户、选择仓库，并确认创建正式订单。
5. 仅在商家确认成功后才创建正式订单、锁定库存并生成订单号。
6. 游客 App 保存临时单据的追踪标识和最近状态，可查询“待商家确认、已转正式订单、已失效/已取消”等状态。

临时单据不等同于订单，不能占用库存、不能参与销售额或客户等级计算，也不能直接触发发货和付款流程。

### 4.3 游客本机数据规则

- 游客订单查询依赖本机安全存储中的追踪标识；清除 App 数据、卸载 App 或更换设备后，游客无法自行恢复这些记录。
- 二维码和追踪标识不得包含完整商品明细、手机号、地址或其他可读敏感信息，只能承载服务端可校验的临时令牌。
- 过期、已被转换或已取消的临时单据不可重复转单；商家扫码确认必须具备幂等保护，防止重复创建订单。
- 商家确认生成正式订单后，订单由商家后台留存；若游客后续绑定登录客户身份，可在后续版本评估将该订单关联到其客户账号。

## 5. 商家端业务规则

### 5.1 商品与库存

- 商品查询展示条码、名称、分类、单位、销售状态和当前仓库库存等必要信息，支持扫码/关键词搜索。
- 库存页面明确区分实际库存、锁定库存和可用库存，其中 `可用库存 = quantity - locked`。
- 入库、出库、调拨、盘点均必须调用后端既有业务接口，并由服务端完成可用库存校验和库存流水记录。
- 高影响操作在客户端必须展示商品、仓库、数量和备注的最终确认信息；失败时以服务端返回结果为准，不做本地补偿写入。

### 5.2 订单处理

- 商家员工只能处理其被授权访问的订单和仓库数据。
- 发货、确认付款、完成、取消均使用既有状态转换规则；客户端根据当前订单状态展示合法操作，服务端负责最终校验。
- 订单详情应展示订单明细、成交单价、备注、状态变更记录和取消原因，避免现场处理时依赖口头信息。
- 扫码解析失败、临时单据过期或已经转单时，必须给出明确结果，不能允许员工手工复用该二维码创建重复订单。

## 6. 数据、缓存与异常处理

- 正式业务数据以服务端为准；App 缓存只用于提升查看体验和保存游客本机购物车/追踪标识，不能作为库存、价格或订单状态的权威来源。
- 客户工作台与商家工作台分别保存缓存、草稿和搜索历史；身份切换或退出时清理另一身份的敏感数据。
- 网络不可用时允许浏览已缓存内容和编辑未提交购物车，但不允许伪造下单、库存变更或订单状态变更。恢复网络后需重新拉取价格、库存和订单状态再提交。
- 对重复点击、网络超时或扫码重复提交的写操作，客户端应使用请求中的幂等标识/结果查询机制；具体协议在后续接口设计中确定。
- 相机、相册和通知等系统权限按功能触发时申请，拒绝权限时提供手动输入或刷新等降级路径。

## 7. 后续接口与模型补充点

现有后台接口覆盖商品、库存、客户和正式订单的核心管理流程。为了支持移动端，后续设计时需补齐或确认以下能力，不能仅由客户端拼装实现：

| 能力 | 目的 |
| --- | --- |
| 客户身份登录与“我的”信息 | 将登录用户与客户档案、等级和本人订单范围可靠关联 |
| 面向客户的商品目录 | 提供适合客户浏览的商品、价格、可售状态与分页搜索接口 |
| 客户自助下单与本人订单查询 | 限制客户只能创建和查询其自身范围内的数据 |
| 临时下单单据与二维码 | 创建、查询、过期、取消、扫码解析和一次性转正式订单 |
| 商家扫码转单 | 选择客户/仓库后创建正式订单，并保证权限、库存校验和幂等性 |
| 移动端消息提醒 | 订单状态变化、待处理订单和库存预警；可在核心流程稳定后再引入 |

接口仍统一挂载在 `/api/v1`，并使用既有 `{ code, message, data }` 响应结构。涉及新表、字段、索引或初始化数据时，后续实施必须按项目约定新增并执行增量 SQL 脚本。

## 8. 一期范围与暂不纳入项

### 一期优先范围

- 一个跨端 App，客户/商家两个隔离工作台及多角色切换。
- 客户侧商品浏览、购物车、已登录客户下单与订单查询。
- 游客临时下单二维码与商家扫码确认转正式订单。
- 商家侧商品查询、库存查询与库存操作、订单查询和合法状态流转、客户查询。
- 基础的登录、鉴权、缓存隔离、扫码和异常反馈。

### 暂不纳入一期

- 第三方在线支付、退款、发票、物流轨迹和配送调度。
- 独立的客户/商家双 App、复杂的多店铺切换和跨租户经营。
- 员工、客户等级、仓库、供应商等全量基础资料维护。
- 离线库存写入、离线订单处理和离线数据冲突合并。
- 营销活动、优惠券、积分、直播/社交分享等零售化能力。

## 9. 验收标准（产品层）

- 用户可在一个 App 内清晰识别自己处于客户还是商家工作台，且无权限时无法进入或操作另一身份的数据。
- 游客可生成二维码让商家接单；同一二维码无论重复扫码还是重复确认，最多生成一张正式订单。
- 游客清除本机数据后不承诺恢复临时记录；正式订单仍由商家系统留存。
- 正式订单状态、库存锁定与释放、会员价计算均以后端规则为准，移动端不改变现有守恒约束。
- 商家端库存和订单写操作具有明确确认、失败反馈和操作记录；客户数据仅在业务所需范围内展示。

## 10. 后续设计决策

在进入实施前，仍需单独确认以下内容：

1. 跨端技术选型（例如 Flutter 或 React Native）及目标平台优先级（iOS、Android）。
2. 客户账号与现有 `Customer` 档案、商家员工账号之间的绑定模型。
3. 游客临时单据的有效期、是否允许游客补充手机号、正式转单后的关联策略。
4. 客户侧“可售库存”的展示策略，以及仓库选择由客户、商家还是系统默认决定。
5. 订单状态通知的渠道与推送权限策略。
