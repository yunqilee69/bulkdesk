# BulkDesk 移动端设计

**日期：** 2026-07-22  
**状态：** 设计已确认，待实现计划  
**范围：** Android、iOS 首发；Harmony 复用业务层并独立验收原生能力。

**2026-07-23 执行说明：** 本轮实现以 iOS 模拟器作为主要原生构建验证目标；Android 与 Harmony 保留代码路径和命令入口，但不在本轮设备验证范围内。

## 1. 目标与边界

BulkDesk 移动端定位为“现场工作台”，而不是 Web 管理后台的缩小版。它服务仓库和配送场景中的高频扫码、采集、执行和留痕，同时支持管理员和仓库管理员在现场为多个客户并行建立待下单清单。

### 1.1 目标

- 让管理员、仓库管理员能在手机端查询客户、扫码查商品、维护多个客户的草稿订单并提交正式订单。
- 让管理员、仓库管理员能用手机完成批量入库、出库、盘点和调拨。
- 让配送员能在手机端完成本人配送任务的照片、手写签名、收款、异常和现场退货闭环。
- 所有价格、库存、权限和订单状态仍以 FastAPI + PostgreSQL 的服务端事务为准。

### 1.2 非目标

- 不将员工、角色、客户等级、会员价、商品基础资料、仓库/供应商维护、价格日志、经营分析和批量后台操作移植到移动端。
- 不支持离线任务队列、离线库存变动或离线提交订单；当前业务确认网络持续可用。
- 不允许移动端临时创建商品或修改客户资料；无法识别的条码回到 Web 维护基础数据。
- 不修改现有订单主状态机：`placed -> shipped -> paid -> completed`，配送签收继续按当前服务转换规则处理。

## 2. 角色和权限

移动端必须基于服务端角色强制授权，隐藏入口只用于体验优化，不能作为鉴权手段。

| 角色 | 移动端范围 |
| --- | --- |
| `admin` | 全部移动端能力；可处理配送、下单、仓库作业、查询全部客户，并可接手草稿。 |
| `warehouse_manager` | 工作台、客户只读查询、商品/库存查询、多客户草稿下单、入库/出库/盘点/调拨，可接手草稿。 |
| `delivery` | 工作台、本人配送任务、任务客户信息、拍照/签名签收、收款、异常、现场退货；不能创建或提交订单，不能修改库存。 |
| `finance` | 首发不提供独立业务工作区；保留 Web 中的财务读取和订单收款管理。 |

客户查询权限为：`admin`、`warehouse_manager` 查询全部客户；`delivery` 仅查询分配给本人的配送客户，且仅显示任务执行所需信息。

## 3. 信息架构

底部导航为“工作台 + 当前角色模块 + 我的”，按角色动态显示。

### 3.1 工作台

所有角色可见，聚合显示：

- 今日待办、待处理订单和待完成配送任务。
- 库存预警（仅具仓库权限者）。
- 配送异常（管理员和配送员按权限过滤）。
- 快捷入口：扫码、创建/打开草稿、开始仓库作业、进入配送任务。

为避免首屏多次分页请求，提供移动端聚合接口 `GET /api/v1/mobile/dashboard`，后端按当前用户角色返回允许展示的摘要和快捷动作。

### 3.2 客户（只读）

独立的客户查询入口，支持按客户名称和手机号搜索；详情包括：

- 客户名称、联系人、电话、地址、备注。
- 客户等级及下单价格依据。
- 订单摘要、配送摘要和最近业务时间。

客户创建、编辑和等级维护继续仅在 Web 完成。配送员的客户查询必须由配送归属校验限制，不能通过直接传入客户 ID 绕过。

### 3.3 订单

对 `admin` 和 `warehouse_manager` 展示：

- 我的草稿：按客户展示当前员工的进行中草稿。
- 可接手草稿：展示可下单员工持有的、允许接手的草稿元数据。
- 新建订单：先选择客户，再搜索商品、输入条码或启动相机扫码加购。
- 提交后的正式订单摘要和详情跳转。

### 3.4 作业

对 `admin` 和 `warehouse_manager` 展示：

- 商品条码扫描和商品/库存查询。
- 批量入库、批量出库、批量盘点、批量调拨。
- 每种作业都先累积为一个批量清单，确认后一次调用现有批量库存接口。

### 3.5 配送

对 `admin` 和 `delivery` 展示：

- 我的配送任务；管理员可按现有权限查看和处理配送任务。
- 客户及订单商品核验、拨打联系人、跳转地图导航（只传递地址，不保存额外定位轨迹）。
- 签收、可选收款、配送异常、现场退货。

### 3.6 我的

所有角色固定可见：账号资料、消息中心、草稿接手记录、设置、退出登录。

## 4. 多客户草稿订单

### 4.1 草稿语义

草稿是独立于 `orders` 的服务端工作区，不生成订单号、不预占库存、不创建库存分配。草稿的业务键为“当前编辑员工 + 客户”，同一员工在同一客户下最多有一份状态为 `editing` 的草稿。

在订单工作区中，用户可以为客户 A 添加商品后不提交，再选择客户 B 创建/打开另一份草稿；两者通过顶部标签切换。标签只负责快速导航，草稿内容始终以服务端数据为准。

### 4.2 草稿数据

新增数据表：

- `order_drafts`：`id`、`customer_id`、`owner_employee_id`、`status`、`remark`、`version`、`created_at`、`updated_at`、`submitted_order_id`、`abandoned_at`。
- `order_draft_items`：`id`、`draft_id`、`product_id`、`quantity`、预估价格快照字段、创建/更新时间；同一草稿内 `product_id` 唯一。
- `order_draft_events`：草稿创建、编辑、接手、放弃、提交成功/失败的审计信息，记录操作者、原/新归属和时间。

`order_drafts` 需要对 `(owner_employee_id, customer_id)` 的进行中草稿建立部分唯一索引。若接手人已经持有同一客户的进行中草稿，首发版本不做草稿合并；接手人需先提交或放弃自己的草稿。

### 4.3 编辑、接手与并发

- 用户选客户时读取或创建自己的 `editing` 草稿。
- 商品按 `product_id` 去重；数量必须大于零；修改商品、数量和备注后自动保存。
- 所有编辑、接手和放弃请求携带 `version`。版本不一致时返回冲突，客户端刷新后明确提示用户，绝不静默覆盖。
- 有下单权限的员工可接手草稿。接手为原子操作：记录审计、变更 `owner_employee_id`、版本递增；原编辑者之后只能查看，不可继续写入。
- 草稿可以主动放弃，但审计记录保留。

### 4.4 价格、库存和提交

草稿页面的金额和单价为预估值。提交时不能信任客户端金额，后端必须在同一事务中：

1. 锁定所涉及的商品、客户和库存记录。
2. 校验商品状态、客户状态、商品去重和数量。
3. 按当前客户等级重算会员价或标准价。
4. 校验可用库存并分配库存。
5. 创建既有 `placed` 正式订单、订单项、库存预留和状态日志。
6. 将草稿标记为 `submitted`，记录 `submitted_order_id` 并递增版本。

任一校验失败时不创建订单、不改变库存、不关闭草稿；API 返回按商品归类的缺货、停售或价格变化原因，用户修正后可再次提交。移动端写请求使用幂等键，避免超时重试产生重复订单。

### 4.5 草稿 API

新增 `/api/v1/order-drafts`，推荐接口如下：

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `GET` | `/order-drafts?scope=mine` | 当前用户的进行中和近期已处理草稿。 |
| `GET` | `/order-drafts?scope=available` | 当前用户可接手的草稿元数据。 |
| `POST` | `/order-drafts` | 传入 `customer_id`，读取或创建当前用户草稿。 |
| `GET` | `/order-drafts/{id}` | 读取草稿详情及商品、客户价格摘要、版本。 |
| `PUT` | `/order-drafts/{id}` | 以完整商品清单、备注和 `version` 保存草稿。 |
| `POST` | `/order-drafts/{id}/takeover` | 接手草稿，携带期望版本。 |
| `POST` | `/order-drafts/{id}/abandon` | 放弃草稿，携带期望版本。 |
| `POST` | `/order-drafts/{id}/submit` | 以幂等键提交；内部复用订单创建服务。 |

订单创建服务应提取为可由既有订单 API 与草稿提交 API 共同调用的业务服务，不复制库存锁定、会员价或状态日志逻辑。

## 5. 仓库扫码作业

扫码支持 EAN/UPC 等条形码与二维码。二维码被识别后需要同时展示原始值和 BulkDesk 业务解析结果，普通二维码不应被误判为扫描失败。

| 作业 | 移动端输入 | 后端行为 |
| --- | --- | --- |
| 批量入库 | 仓库、供应商（可选）、商品数量、成本价（可选）、备注 | 复用 `/inventory/stock-in/batch`。 |
| 批量出库 | 仓库、商品数量、备注 | 复用 `/inventory/stock-out/batch`；提交前展示可用库存，后端仍是最终校验。 |
| 批量盘点 | 仓库、商品实际数量、备注 | 复用 `/inventory/stocktake/batch`，明确展示盘盈/盘亏差异。 |
| 批量调拨 | 来源仓库、目标仓库、商品数量、备注 | 复用 `/inventory/transfer/batch`；来源和目标必须不同。 |

扫描到未知条码、失效商品或无权限仓库时，客户端不允许把项目加入清单，并显示可操作原因。提交使用前端提交锁，防止重复点击；后端继续保持事务校验。

## 6. 配送、签收、收款和退货

### 6.1 配送任务

复用当前配送任务、权限和状态规则：配送员只能处理本人状态为 `delivering` 的任务；管理员按现有权限处理和改派。详情显示客户收货信息、订单商品、历史异常和可退商品。

### 6.2 签收凭证

移动端签收流程要求：

- 收货人姓名。
- 至少一张现场照片。
- 直接手写产生的 PNG 签名。
- 可选备注。

图片和签名先上传到现有 `/api/v1/upload`，上传成功的 URL 再提交签收。客户端需要在拍摄后检查 MIME 类型、尺寸和 10 MB 服务端上限，必要时压缩或引导重新拍摄。

当前配送模型只有 `proof_image_urls`，必须新增可空字段 `signature_image_url`，并扩展 `OrderDeliverySignRequest`、配送详情响应和 Web 配送详情展示。为了兼容历史 Web 签收记录，数据库字段保持可空；移动端自己的签收校验必须要求现场照片和签名。签收业务服务仍由现有 `sign_delivery` 统一处理，不新建平行状态机。

### 6.3 收款、异常和退货

- 收款为签收内的可选动作；开启后必须填写实收金额并上传至少一张付款凭证，复用当前支付完成逻辑。
- 异常类型沿用：客户不在、客户拒收、联系方式错误、其他；“其他”必须填写说明。记录异常不替代后续签收或改派处理。
- 现场退货由配送任务加载可退商品，选择数量、原因、品相和是否入库后，调用既有退货单创建接口。

## 7. 客户端技术架构

### 7.1 分层

在现有 React Native POC 基础上建立以下边界：

- **App shell**：登录恢复、动态导航、角色路由守卫、全局错误页和权限不足页。
- **Feature modules**：`dashboard`、`customers`、`orders`、`inventory`、`delivery`、`profile`，每个模块拥有页面、表单模型和调用用例。
- **API layer**：复用现有 `createApiClient`；为每个业务模块提供强类型 API 封装，统一添加认证、超时、幂等键和错误映射。
- **Server-state layer**：缓存查询结果、失效关联查询、处理加载/刷新/错误状态。在线模式下不建立离线写入队列。
- **Platform adapters**：保持现有扫码、拍照、签名、图片校验、上传和安全存储的抽象；Android/iOS/Harmony 在适配器内部差异化实现。

移动端令牌仅放入平台安全存储，不能放入普通 `AsyncStorage`、日志、URL 或图片元数据。退出登录时清空会话和受保护查询缓存。

### 7.2 原生能力

| 能力 | 当前 POC | 正式应用要求 |
| --- | --- | --- |
| 扫码 | Vision Camera 真机已验证条形码和二维码 | 权限拒绝、相机不可用、重复结果节流、未知条码提示。 |
| 拍照 | Vision Camera 真机预览与拍摄链路 | 图片压缩/大小校验、上传进度、重试与删除失败文件。 |
| 手写签名 | PanResponder 连续笔迹和 PNG 导出 | 空签名禁用提交、导出后上传、将 URL 绑定配送签收。 |
| 安全存储 | 已有抽象 | Android Keystore、iOS Keychain、Harmony 对等安全能力分别验收。 |

Android 为首发真机验收平台；iOS 同步构建和回归；Harmony 复用 TypeScript 业务层，但相机、签名、安全存储和上传分别验收。Harmony 任一原生能力未通过时不阻塞 Android/iOS 发布。

## 8. 后端改造和数据迁移

实施时需新增：

1. 草稿订单模型、schema、service、API 和测试。
2. 订单创建服务公共化，以保证草稿提交和 Web 创建订单的库存、价格与状态规则完全一致。
3. 移动工作台聚合、条码商品摘要和客户移动详情接口；只返回当前角色所需字段。
4. `order_deliveries.signature_image_url` 字段、签收请求/响应和 Web 详情展示。
5. 写操作的版本冲突响应和幂等键去重策略。

所有数据库变更必须生成按日期归档的 `backend/migrations/incremental/YYYY-MM-DD_简要说明.sql`，并在开发库通过 DBX `postgres` 执行后检查 schema/数据确认生效。迁移与 Alembic 策略保持现有项目约定，不只提交脚本。

## 9. 接口安全与可观测性

- 每个写接口使用现有 `AdminUser`、`WarehouseUser`、`DeliveryUser` 或明确的角色依赖，且服务层二次校验草稿归属、配送归属和客户可见范围。
- 草稿接手、草稿提交、库存操作、配送签收、收款和退货均记录操作者、时间、关联实体和必要的前后状态。
- 对提交订单、库存批量操作、签收和上传失败提供可追踪请求 ID；客户端在提示中展示可供支持人员排查的标识。
- 不在客户端日志输出令牌、客户完整敏感信息、签名图片二进制或收款凭证 URL。

## 10. 分期实施与验收

### P0：应用骨架与只读查询

- 登录/会话恢复、动态导航、统一 API/错误层。
- 工作台、客户只读查询、商品/条码查询。
- Android、iOS 构建与基础角色权限验收。

### P1：多客户移动下单

- 草稿表、迁移、API、接手审计、版本冲突和幂等提交。
- 多客户标签、扫码/搜索加购、自动保存、客户价格预估、提交失败逐项修正。
- 验收：多客户切换不丢数据；同一草稿并发编辑不静默覆盖；提交只在成功时锁定库存并生成一张正式订单。

### P2：仓库扫码作业

- 批量入库、出库、盘点、调拨工作流和条码扫描。
- 验收：每种操作正确写入现有库存流水；重复提交不造成重复变动；未知条码和库存不足均有明确反馈。

### P3：配送闭环

- 配送任务、拍照、手写签名、签收、收款、异常、现场退货。
- 配送签名字段迁移和 Web 详情展示。
- 验收：真机连续手写签名可上传并回显；照片、签名和收款凭证成功上传后才允许签收；订单状态遵循现有服务端状态机。

## 11. 测试策略

- **后端单元/服务测试**：草稿唯一性、版本冲突、接手权限、草稿提交原子性、价格重算、库存不足回滚、签名字段兼容性。
- **API 集成测试**：角色越权、配送归属、客户可见性、幂等键、批量库存操作和上传约束。
- **移动端单元测试**：草稿标签状态、扫码去重、价格/数量格式、空签名、上传失败和 API 错误映射。
- **真机回归**：Android 必测扫码、拍照、连续签名和上传；iOS 同样回归；Harmony 按原生能力矩阵单独记录通过/阻断状态。
- **完成验证**：按项目约定执行后端 `uv run pytest`、编译检查，以及移动端 `npm test -- --runInBand`、`npm run typecheck`、`npm run lint`、Android/iOS 构建。任何无关既有失败必须保留输出并说明。

## 12. 风险与决策

| 决策/风险 | 处理方式 |
| --- | --- |
| 草稿价格与提交价格不同 | 草稿仅显示预估，提交时服务端重算并回传差异。 |
| 多人编辑或接手覆盖内容 | 版本号 + 原子接手 + 审计，不使用最后写入覆盖。 |
| 网络超时导致重复订单 | 提交请求使用幂等键，后端持久化去重结果。 |
| 真机媒体文件超过服务端限制 | 客户端拍摄后校验/压缩，服务端继续强制 10 MB 限制。 |
| Harmony 原生库兼容性 | 通过平台适配器隔离，按能力独立验收，不影响 Android/iOS 首发。 |
