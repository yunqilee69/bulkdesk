# 订单配送管理设计

## 背景

现有销售订单已经支持 `placed -> shipping -> stocked_out -> delivered_unpaid -> completed` 履约流程。订单确认出库后会扣减库存，但系统只记录执行订单状态操作的账号，无法回答实际由哪名员工送货、客户由谁签收，也无法按配送员查看当前负责的订单。

配送场景不需要车次、车辆、线路或配送批次管理。系统只需在订单出库时绑定配送员，将同一配送员当前尚未签收的订单临时聚合展示；订单签收后自动进入历史查询。

## 目标

- 订单确认出库时必须绑定一名启用员工，并保存本次收货信息快照。
- 按配送员查看其当前负责的订单数量、客户数量、商品件数、配送金额和异常数量。
- 记录实际签收人、可选签收照片和签收备注。
- 支持配送员本人或管理员登记签收与配送异常。
- 支持管理员在签收前改派配送员，并永久保留改派历史。
- 签收完成后自动归档配送记录，不增加人工归档操作。
- 保持现有订单状态机和库存守恒规则不变。

## 非目标

- 不建立配送任务、配送批次或配送车次。
- 不管理配送车辆、线路、里程、运费或导航。
- 不支持一个订单拆分给多个配送员或分批签收。
- 不建设独立配送员移动端；配送员使用现有员工账号访问响应式管理页面。
- 不提供已签收记录的撤销或修改能力；首版纠错通过管理员线下核实和后续专门审计功能处理。

## 领域规则

### 配送关系

- 一个销售订单最多对应一条配送记录。
- 一个配送记录只关联一个销售订单和一名当前配送员。
- 任意启用员工均可被选择为配送员，不新增专门的配送员角色或可配送标记。
- 同一配送员当前负责的多个订单通过查询实时聚合，不保存为任务、批次或车次。
- 一个订单不允许拆分配送，也不记录部分签收。

### 配送状态

配送记录只有两个状态：

- `delivering`：配送中，订单已经确认出库但尚未成功签收。
- `signed`：已签收，同时视为配送归档。

配送异常不会改变配送状态。处于 `delivering` 的记录可以多次登记异常、成功签收或由管理员改派。

### 收货快照

确认出库时默认从客户资料带入联系人、联系电话和地址，并允许操作员针对本订单修改。三项信息均为必填，保存后不反写客户资料，也不随客户资料后续修改而变化。

### 签收凭证

- 实际签收人姓名必填。
- 签收照片可选，允许上传多张照片。
- 签收备注可选。
- 签收照片可以是签字单、货物交接照片或其他现场凭证。
- 系统同时记录签收登记操作人和登记时间，实际签收人与系统操作人是不同概念。

### 配送异常

首版异常类型包括：

- `customer_absent`：客户不在。
- `customer_refused`：客户拒收。
- `invalid_contact`：地址或联系方式有误。
- `other`：其他。

登记异常时异常类型必填，说明可选；选择 `other` 时说明必填。异常不会触发订单取消、退货或归档。

### 配送改派

- 仅管理员可以改派。
- 仅 `delivering` 状态允许改派。
- 新配送员必须是启用员工，且不能与当前配送员相同。
- 改派原因可选。
- 系统保存原配送员、新配送员、操作人、时间和原因。

## 订单状态与事务边界

### 确认出库

订单从 `shipping` 转为 `stocked_out` 时，请求必须同时提交配送员和收货快照。以下操作必须在同一数据库事务中完成：

1. 锁定并校验订单及最终库存分配。
2. 按现有规则扣减实际库存并释放库存预占。
3. 将订单状态更新为 `stocked_out`。
4. 创建状态为 `delivering` 的配送记录。
5. 写入首次分配事件。

任一步骤失败时整笔事务回滚，不允许出现库存已扣减但配送记录不存在的订单。

### 成功签收

签收时以下操作必须在同一数据库事务中完成：

1. 锁定配送记录与对应订单。
2. 校验配送记录为 `delivering`、订单为 `stocked_out`。
3. 校验当前操作人有权处理该记录。
4. 保存实际签收人、照片、备注、签收时间和登记操作人。
5. 写入成功签收事件，并将配送记录更新为 `signed`。
6. 按现有状态机将订单更新为 `delivered_unpaid`，记录订单送达操作信息。

任一步骤失败时整笔事务回滚。重复请求不得产生重复签收事件，也不得重复推进订单状态。

### 配送异常与改派

配送异常和改派均不改变订单状态。操作时锁定配送记录，校验记录仍为 `delivering`，防止与签收并发覆盖。

## 数据模型

### `order_deliveries`

| 字段 | 说明 |
| --- | --- |
| `id` | UUID 主键 |
| `order_id` | 销售订单 ID，唯一外键 |
| `delivery_employee_id` | 当前配送员 ID |
| `delivery_employee_name` | 当前配送员姓名快照 |
| `status` | `delivering` 或 `signed` |
| `recipient_name` | 本次收货联系人快照 |
| `recipient_phone` | 本次联系电话快照 |
| `delivery_address` | 本次配送地址快照 |
| `assigned_at` | 首次分配时间 |
| `assigned_by_id` | 首次分配操作人 ID |
| `assigned_by_name` | 首次分配操作人姓名快照 |
| `signer_name` | 实际签收人姓名，可空 |
| `proof_image_urls` | 签收照片 URL 列表，可空 |
| `sign_remark` | 签收备注，可空 |
| `signed_at` | 签收时间，可空 |
| `signed_by_id` | 签收登记操作人 ID，可空 |
| `signed_by_name` | 签收登记操作人姓名快照，可空 |
| `created_at` / `updated_at` | 创建与更新时间 |

约束与索引：

- `order_id` 唯一约束，保证一单一条配送记录。
- `delivery_employee_id, status` 联合索引，支持当前配送按人员聚合。
- `status, signed_at` 联合索引，支持归档分页和日期筛选。

### `order_delivery_events`

| 字段 | 说明 |
| --- | --- |
| `id` | UUID 主键 |
| `delivery_id` | 配送记录 ID |
| `event_type` | `assigned`、`reassigned`、`exception`、`signed` |
| `from_employee_id` / `from_employee_name` | 原配送员及姓名快照，可空 |
| `to_employee_id` / `to_employee_name` | 新配送员及姓名快照，可空 |
| `exception_type` | 配送异常类型，可空 |
| `remark` | 改派原因、异常说明或签收备注，可空 |
| `operator_id` | 系统操作员工 ID |
| `operator_name` | 系统操作员工姓名快照 |
| `created_at` | 事件发生时间 |

事件表只追加、不更新、不删除，用于展示完整配送时间线。事件类型与相关字段组合由服务层校验。

## 权限设计

### 管理员

- 在订单确认出库时选择任意启用员工为配送员。
- 查看全部当前配送与配送归档。
- 登记任意配送记录的签收或异常。
- 改派任意未签收配送记录。

### 普通员工

- 只查看当前分配给自己的配送记录和自己的配送归档。
- 只对当前分配给自己的 `delivering` 记录登记签收或异常。
- 不允许改派配送员，也不能处理其他员工的记录。

员工被禁用后不能接收新配送且不能登录。其未完成配送由管理员登记或改派，历史记录保留员工 ID 与姓名快照。

## API 设计

所有接口挂载在 `/api/v1` 下，并使用统一响应结构。

### 调整订单确认出库

现有确认出库接口请求体增加：

```json
{
  "delivery_employee_id": "uuid",
  "recipient_name": "李四",
  "recipient_phone": "13800000000",
  "delivery_address": "客户本次收货地址"
}
```

配送字段与库存出库校验失败时，统一拒绝确认出库。

### 配送查询

- `GET /api/v1/deliveries/current`
  - 管理员返回按配送员分组的当前配送数据。
  - 普通员工只返回自己的分组。
  - 支持订单号、客户关键字、配送员和是否存在异常筛选。
  - 分组汇总订单数、客户数、商品件数、订单总金额和存在异常的订单数。
- `GET /api/v1/deliveries/archive`
  - 分页返回 `signed` 记录。
  - 支持配送员、订单号、客户、签收人和签收日期范围筛选。
  - 普通员工只能查询自己的归档。
- `GET /api/v1/deliveries/{delivery_id}`
  - 返回订单摘要、配送快照、签收凭证和按时间排序的事件列表。
  - 普通员工只能查看自己当前负责或历史负责并最终完成的配送记录；管理员可查看全部。

### 配送操作

- `PUT /api/v1/deliveries/{delivery_id}/reassign`
  - 管理员接口。
  - 请求包含新配送员 ID 和可选原因。
- `POST /api/v1/deliveries/{delivery_id}/exceptions`
  - 当前登录员工接口。
  - 请求包含异常类型和说明。
- `PUT /api/v1/deliveries/{delivery_id}/sign`
  - 当前登录员工接口。
  - 请求包含签收人姓名、签收照片 URL 列表和可选备注。

## 前端设计

### 菜单与路由

新增“配送管理”菜单与页面，页面包含“当前配送”和“配送归档”两个页签。

### 当前配送

管理员按配送员展示可展开汇总卡片；普通员工只展示自己的汇总与订单列表。每个配送员汇总显示：

- 待签收订单数。
- 涉及客户数。
- 商品总件数。
- 配送订单总金额。
- 存在配送异常的订单数。

订单明细展示订单号、客户、联系人、联系电话、配送地址、商品摘要、金额、出库时间和最近一次异常。配送员与管理员可登记签收和异常，管理员额外显示改派操作。

### 配送归档

使用分页表格展示已签收记录，支持配送员、订单号、客户、签收人和签收日期筛选。详情展示：

- 订单和商品摘要。
- 出库时收货信息快照。
- 配送员及改派历史。
- 配送异常历史。
- 实际签收人、签收时间、照片和备注。
- 每次操作的系统操作人。

### 订单页面调整

- 确认出库弹窗在原有出库确认内容基础上增加配送员、联系人、联系电话和配送地址，四项均必填。
- 客户联系人、电话和地址默认带入出库弹窗，允许本次调整。
- 订单详情在进入 `stocked_out` 后展示“配送信息”区域，包括当前配送员、配送快照、异常摘要和最终签收信息。
- 已出库订单不再直接使用仅记录操作账号的“确认送达”入口，送达必须通过配送签收操作完成。

## 错误处理

- 配送员不存在或已禁用：拒绝出库或改派。
- 联系人、电话或地址为空：拒绝确认出库。
- 订单不是 `shipping`：拒绝携带配送信息确认出库。
- 配送记录不是 `delivering`：拒绝改派、异常登记和签收。
- 订单不是 `stocked_out`：拒绝异常登记和签收，并记录服务端一致性错误。
- 普通员工处理非本人记录：返回无权限错误，不泄露记录详情。
- 上传图片失败：签收表单不提交，允许用户重试或移除失败图片。
- 并发签收、签收与改派并发：以后获得锁的请求重新校验状态并失败返回，不覆盖已完成结果。

## 测试与验证

### 后端测试

- 确认出库缺少配送员、联系人、电话或地址时失败。
- 已禁用员工不能接收新配送。
- 库存扣减、订单状态更新、配送记录和首次事件在同一事务成功或回滚。
- 一个订单不能创建重复配送记录。
- 管理员可查看和操作全部记录，普通员工只能查看和操作自己的记录。
- 改派更新当前配送员并保留原、新配送员事件快照。
- 同一记录可连续登记多次配送异常，异常不改变订单和配送状态。
- 成功签收保存凭证、写入事件并将订单推进到 `delivered_unpaid`。
- 重复签收、已签收后改派、并发签收均被拒绝。
- 当前配送聚合的订单数、去重客户数、商品件数、订单金额和异常订单数正确。
- 归档分页、筛选和普通员工数据范围正确。

### 前端测试

- 确认出库表单正确带入客户信息并校验四个配送字段。
- 当前配送汇总数据映射和异常计数展示正确。
- 管理员与普通员工操作按钮符合权限范围。
- 签收表单校验签收人，正确收集成功上传的照片 URL。
- 异常类型为 `other` 时要求填写说明。
- 改派、异常和签收成功后刷新当前配送与订单详情。
- 配送归档筛选参数和分页字段与后端契约一致。

## 验收标准

1. 订单不选择有效配送员或缺少完整收货信息时不能确认出库。
2. 确认出库后，管理员能在配送管理中按配送员看到该订单及正确汇总。
3. 被分配的普通员工登录后只能看到并处理自己的配送订单。
4. 配送异常可以多次登记，订单保持 `stocked_out` 且仍在当前配送中。
5. 管理员改派后，新配送员立即看到订单，原配送员不能继续处理，历史保留完整交接记录。
6. 登记签收后，配送记录自动进入归档，订单进入 `delivered_unpaid`。
7. 配送归档能查询实际配送员、实际签收人、签收照片、异常和改派历史。
8. 库存、订单和配送数据在失败或并发操作下保持一致。
