# 订单跨仓库存预占与发货设计

## 目标

订单创建者只选择客户、商品和数量。系统在创建订单时自动跨仓分配并锁定库存；实际发货时，操作员可以调整每个商品对应的仓库和数量，并按最终分配扣减库存。

## 核心规则

- `orders` 不再保存仓库字段。
- 同一订单明细可以由多个仓库共同预占和发货。
- 创建订单时必须完成全部商品的库存预占；任一商品库存不足时整单回滚。
- 自动分配优先默认仓库，然后按可用库存从高到低分配。
- 发货时允许重新分配仓库，但每条订单明细的最终分配总量必须等于订购数量。
- 本阶段保持整单一次性发货，不支持部分发货。
- 所有库存检查、预占调整和实际扣减必须在数据库事务和行锁内完成。

## 数据模型

新增 `order_inventory_allocations` 表：

- `id`：UUID 主键。
- `order_id`：订单 ID。
- `order_item_id`：订单明细 ID。
- `product_id`：商品 ID。
- `warehouse_id`：仓库 ID。
- `quantity`：该仓库分配数量。
- `status`：`reserved`、`shipped`、`released`、`returned`。
- `created_at`、`updated_at`：时间戳。

约束：

- `(order_item_id, warehouse_id)` 唯一。
- `quantity > 0`。
- 订单、订单明细、商品和仓库使用外键约束。

删除 `orders.warehouse_id`，不保留兼容字段。迁移时根据旧订单的仓库和订单明细回填分配记录，然后删除原字段。

## 创建订单

后端验证客户、商品状态和数量后，按固定顺序锁定相关库存行。每个商品从默认仓库开始分配，不足部分继续从其他启用仓库分配。分配成功后增加 `inventory.locked`，创建订单明细及 `reserved` 分配记录。

创建接口不再接收 `warehouse_id`。响应中的订单明细包含当前库存分配，供详情页和发货页使用。

## 发货

发货请求提交每个订单明细的最终仓库分配。后端锁定原预占和目标库存行，先在计算中释放原订单预占，再验证新分配是否可用，之后原子性地更新锁定量、扣减实际数量并将最终分配标记为 `shipped`。

每个仓库分别生成一张 `order_deduction` 库存流水。任一校验失败时整个事务回滚。

## 取消

- `placed`：释放 `reserved` 分配对应的锁定库存，状态改为 `released`。
- `shipped` 或 `paid`：根据 `shipped` 分配退回原仓库，状态改为 `returned`，并按仓库生成退货流水。

## 前端

- 新建订单移除仓库选择、仓库库存加载和单仓可用量限制。
- 发货操作改为分配弹窗，默认展示系统预占结果。
- 每个商品允许添加多个“仓库 + 数量”行。
- 展示仓库可用数量，并校验商品分配合计。
- 订单详情展示预占或实际发货仓库分配。

## 本阶段不包含

- 部分发货和多张发货单。
- 缺货订单、延期分配或预售。
- 自动释放超时未处理订单。
- 客户端下单页面；本次只确保接口不要求客户选择仓库。
