# 退货来源追踪与多角色权限设计

## 背景

当前系统的退货单不关联原销售订单，允许人工填写商品、数量和退货单价。该方式无法证明商品真实售出，也无法限制累计退货数量，存在库存和客户累计消费被错误调整的风险。

员工目前只有单值 `admin`、`normal` 角色，无法表达仓储运营、配送和财务职责，也无法满足同一员工兼任多个岗位的实际场景。

本次设计同时处理以下事项：

1. 退货必须追踪到原销售订单明细。
2. 配送员可在当前配送现场办理同一客户历史订单的退货。
3. 员工支持多角色，权限按角色并集计算。
4. 标准售价和会员价禁止为零。
5. 客户等级当前仅人工维护，不自动升级或降级。

## 目标

- 防止未销售商品、超量商品或伪造价格进入退货流程。
- 保留原订单成交金额，同时独立记录退货金额和应收净额。
- 支持配送员在当前客户现场处理该客户以前订单的退货。
- 用固定业务角色清晰隔离管理、仓储运营、配送和财务权限。
- 消除零元会员价造成的前后端计价不一致。
- 明确会员等级自动变更不属于当前实现范围。

## 非目标

- 本次不实现多次收款、应收账龄和坏账处理。
- 本次不实现实际退款流水、退款渠道或退款审批。
- 本次不实现采购订单、应付账款和采购退货。
- 本次不实现动态角色权限配置后台。
- 本次不实现赠品业务。
- 本次不实现客户等级自动升级、降级或保级规则。

## 领域规则

### 办理配送记录

`handling_delivery` 表示配送员本次到达客户现场所执行的配送任务。它只用于确认办理权限、当前客户和操作审计，不代表退回商品一定来自该配送任务对应的订单。

普通配送员办理退货时必须满足：

- 配送记录状态为 `delivering`。
- 配送记录的当前配送员是当前登录员工。
- 当前员工拥有 `delivery` 角色。

管理员可以代办退货，不要求本人是当前配送员，但仍必须选择一个客户和真实来源订单明细。

### 来源订单明细

每条退货明细必须关联一个 `source_order_item`。来源订单可以与办理配送记录对应的订单不同，但必须满足：

- 来源订单客户与办理配送记录客户相同。
- 来源订单状态为 `stocked_out`、`delivered_unpaid` 或 `completed`。
- 商品、条码、成交单价由来源订单明细读取，前端不得自行提交或修改。
- 同一来源订单明细的全部有效退货数量之和不得超过原销售数量。
- 已作废退货单的明细不计入累计退货数量。

一张退货单可以包含同一客户多个历史订单的明细，因此来源关系记录在退货明细上，而不是只记录在退货单头上。

### 退货状态

退货保持单阶段处理：

```text
completed -> voided
```

- 配送员现场提交成功后，退货单直接进入 `completed`。
- 不增加 `pending`、审核或仓库确认状态。
- 已完成退货单只能由 `admin` 作废。

### 退货金额与订单净额

- 每条退货明细金额为来源订单明细的 `unit_price * return_quantity`。
- 原订单 `total_amount` 保持不变，继续作为原始成交快照。
- 每张来源订单维护累计有效退货金额 `returned_amount`。
- 应收净额通过 `net_amount = total_amount - returned_amount` 计算。
- 退货后确认收款时，实收金额不得超过 `net_amount`。
- 来源订单尚未完成收款时，退货只减少订单应收净额，不调整客户累计消费。
- 来源订单已经 `completed` 时，退货同时按退货金额冲减客户 `total_spent`。
- 作废退货单时，反向恢复来源订单 `returned_amount` 和实际冲减的客户累计消费。

同一退货单可能关联多张来源订单，服务层必须按来源订单分别汇总和更新退货金额。

### 库存处理

- 每条退货明细继续记录商品状况、退货原因、是否入库和入库仓库。
- 选择入库时必须选择启用仓库。
- 不入库时不得保留仓库 ID。
- 创建退货单时立即增加所选仓库库存，并生成 `customer_return_in` 流水。
- 作废退货单时必须保证原入库数量仍有足够可用库存扣回，并生成 `customer_return_void_out` 流水。
- 商品停售不影响历史销售商品退货。

## 数据模型

### 员工角色

员工与角色改为多对多关系。固定角色代码如下：

- `admin`
- `warehouse_manager`
- `delivery`
- `finance`

新增员工角色关联表，至少包含：

- `employee_id`
- `role`
- `created_at`
- `(employee_id, role)` 唯一约束

删除单值 `employees.role` 作为业务权限来源。现有数据迁移规则：

- `admin` 迁移为 `admin`。
- `normal` 迁移为 `warehouse_manager`。

登录身份和当前员工接口返回 `roles: string[]`。权限判断使用角色集合，不再依赖单个角色字段。

### 订单

订单新增：

- `returned_amount Numeric(12, 2) NOT NULL DEFAULT 0`

数据库约束：

- `returned_amount >= 0`
- `returned_amount <= total_amount`

`net_amount` 优先作为响应计算字段，不单独持久化，避免与 `total_amount`、`returned_amount` 产生重复数据不一致。

### 退货单

退货单新增：

- `handling_delivery_id`，关联办理现场的配送记录。

管理员代办时仍记录所选择的办理配送记录；如果未来需要支持完全脱离配送任务的后台退货，再单独设计管理员例外来源，不在本次范围内。

### 退货明细

退货明细新增：

- `source_order_item_id`，关联原销售订单明细。

保留商品名称、条码和成交单价快照，便于历史展示，但这些字段只能由后端从来源订单明细生成。

## 固定角色权限

员工拥有多个角色时，权限取并集。`admin` 拥有全部权限。

### 仓库管理员 `warehouse_manager`

- 创建商品，维护商品名称、条码、分类、品牌、规格、单位、图片、状态和库存预警值。
- 维护分类、品牌、仓库和供应商资料。
- 修改商品成本价、标准售价和会员价，并记录价格变更日志。
- 执行入库、手工出库、调拨和盘点。
- 创建订单、开始发货和在出库前取消订单。
- 调整订单分仓并确认销售出库。
- 可以查看客户资料用于订单办理，但不能创建或修改客户资料。

### 配送员 `delivery`

- 查看本人当前和历史配送任务。
- 对本人当前任务登记配送异常和签收。
- 对本人当前任务确认现场收款并上传付款凭证。
- 基于本人当前配送任务，办理同一客户历史订单商品退货。
- 被改派后不能再操作原配送任务。
- 不能作废退货单或撤销已确认收款。

### 财务 `finance`

- 查看全部订单收款信息、付款凭证、退货金额和应收净额。
- 为后续退款、冲正、对账和应收管理预留权限。
- 本次不新增退款和冲正写接口。

### 管理员 `admin`

- 拥有全部业务权限。
- 创建和维护客户资料。
- 管理员工及其角色集合。
- 作废已完成退货单。
- 处理其他角色无法执行的异常业务操作。

## 价格规则

- 商品标准售价必须大于零。
- 会员价必须大于零。
- 成本价可以为零。
- 前端表单和后端 schema 必须使用相同约束。
- 后端读取会员价时必须显式判断是否为 `None`，不得使用真假值回退。
- 赠品不能通过零元会员价表示，后续设计独立赠品功能。

## 会员等级规则

- 当前客户等级只作为人工维护的业务记录。
- `min_spent` 只记录等级门槛，不触发自动升级或降级。
- 订单完成和退货处理可以更新 `total_spent`，但不得自动修改 `level_id`。
- 当前版本不自动写入等级变更日志。
- 自动升级、自动降级、保级周期和等级变更日志属于后续功能。

项目协作规则、客户模块文档和功能清单必须使用上述一致描述，不再声明当前订单完成后自动更新客户等级。

## 接口契约

### 当前员工

当前员工响应从单值角色调整为：

```json
{
  "id": "employee-id",
  "username": "employee",
  "roles": ["warehouse_manager", "delivery"]
}
```

### 创建退货单

请求结构调整为：

```json
{
  "handling_delivery_id": "delivery-id",
  "items": [
    {
      "source_order_item_id": "order-item-id",
      "quantity": 2,
      "condition": "normal",
      "return_reason": "客户退回",
      "remark": null,
      "should_stock_in": true,
      "warehouse_id": "warehouse-id"
    }
  ],
  "remark": null
}
```

请求不再接受客户端提交的 `customer_id`、`product_id` 或 `unit_price`。客户由办理配送记录确定，商品和成交单价由来源订单明细确定。

### 历史可退明细

新增查询接口，用于配送员按当前配送任务查询同一客户的历史可退商品：

```text
GET /api/v1/deliveries/{delivery_id}/returnable-items
```

返回内容至少包含：

- 来源订单 ID、订单号和状态。
- 来源订单明细 ID。
- 商品 ID、名称和条码。
- 原销售数量、累计已退数量和当前可退数量。
- 原成交单价。

普通员工只能查询本人当前配送任务；管理员可以查询任意符合条件的配送任务。

## 前端交互

### 员工管理

- 员工创建和编辑表单将单选角色改为多选角色。
- 至少选择一个角色。
- 列表以标签形式展示全部角色。
- 前端菜单权限根据角色集合判断。

### 退货办理

- 配送任务详情提供“办理退货”入口。
- 点击后按当前配送任务加载同一客户历史可退明细。
- 配送员只能选择后端返回的来源明细。
- 页面展示原订单号、原成交单价、已退数量和剩余可退数量。
- 配送员填写退货数量、商品状况、退货原因、是否入库和入库仓库。
- 提交成功后直接显示已完成退货单。
- 管理员仍可在退货单页面查看和作废退货记录。

### 价格表单

- 标准售价和会员价输入框最小值改为大于零的最小货币单位。
- 仓库管理员可以创建商品并维护成本价、标准售价和会员价。
- 客户创建和编辑入口仅管理员可见；仓库管理员保留客户只读选择能力以创建订单。

## 并发与事务

创建退货单必须在一个事务内完成：

1. 锁定办理配送记录及对应当前订单。
2. 校验当前用户角色和配送任务归属。
3. 按固定顺序锁定全部来源订单明细和来源订单。
4. 查询并校验累计有效退货数量。
5. 按固定顺序锁定需要变更的库存行。
6. 创建退货单和退货明细。
7. 更新来源订单 `returned_amount`。
8. 更新库存及库存流水。
9. 按实际影响更新客户累计消费。

任一步骤失败时整笔退货回滚，不允许出现退货单、库存、订单退货金额或客户累计消费部分成功。

作废退货单同样必须在单一事务内执行完整反向操作。

## 数据迁移

需要新增按日期归档的增量 SQL，并在开发库使用 DBX `postgres` 连接执行和验证。迁移至少包括：

- 新建员工角色关联结构。
- 将现有 `admin`、`normal` 角色迁移到关联表。
- 调整员工单值角色字段。
- 为订单增加 `returned_amount` 和检查约束。
- 为退货单增加 `handling_delivery_id`。
- 为退货明细增加 `source_order_item_id`。
- 为新增外键和查询条件增加必要索引。
- 验证所有员工至少拥有一个角色。
- 验证订单退货金额均处于合法范围。

## 测试范围

### 后端

- 零元标准售价和会员价被拒绝。
- 会员价回退只在会员价不存在时发生。
- 现有单角色员工正确迁移为多角色数据。
- 多角色权限取并集。
- 普通员工无法越权执行其他角色操作。
- 配送员可以对本人当前任务客户的历史订单办理退货。
- 配送员不能对其他客户或其他配送员任务办理退货。
- 来源订单明细状态和客户不匹配时拒绝退货。
- 多次退货累计数量不能超过原销售数量。
- 并发退货不能突破可退数量。
- 未收款订单退货只更新应收净额。
- 已完成订单退货同时冲减客户累计消费。
- 停售商品可以按历史订单退货。
- 作废退货单完整恢复订单金额、库存和客户累计消费。
- 退货事务中任一步骤失败时全部回滚。

### 前端

- 员工角色支持多选和正确回显。
- 不同角色只展示允许的菜单和操作按钮。
- 配送任务可以打开历史可退商品选择界面。
- 退货页面不允许手工修改商品和成交单价。
- 剩余可退数量正确限制输入。
- 仓库管理员和管理员可以维护全部商品价格，其他角色只能查看。
- 标准售价和会员价拒绝零值。

## 验收标准

- 任何有效退货明细都能追踪到原订单明细。
- 任何来源订单明细的累计有效退货数量都不超过原销售数量。
- 配送员只能在本人当前配送任务中处理同一客户退货。
- 原订单金额保持不变，累计退货金额和应收净额准确。
- 库存、订单退货金额和客户累计消费在创建及作废后保持守恒。
- 员工可以拥有多个固定业务角色，前后端权限判断一致。
- 标准售价和会员价不能保存为零。
- 订单完成和退货不会自动修改客户等级。
