# 商品列表与库存选品搜索条件设计

**日期：** 2026-07-17
**状态：** 已确认，待实施

## 目标

优化商品列表的搜索条件，并让库存出入库、调拨和盘点共用的商品选择弹窗复用同一套服务端搜索能力，避免一次性加载全部商品。

## 范围

### 商品列表

商品列表搜索区提供以下条件：

- 商品关键词：一个输入框，模糊匹配商品名称或商品简称，任一字段命中即可。
- 条形码：一个独立输入框，模糊匹配条形码。
- 分类：按 `category_id` 精确筛选。
- 品牌：按 `brand_id` 精确筛选。
- 销售状态：按 `status` 筛选在售或停售。
- 成本价范围：可单独输入下限或上限，分别映射为 `min_cost_price` 和 `max_cost_price`。
- 售价范围：可单独输入下限或上限，分别映射为 `min_standard_price` 和 `max_standard_price`。

价格范围比较均包含边界。例如成本价范围 `10` 至 `20` 会匹配成本价等于 `10`、等于 `20` 以及处于两者之间的商品。

### 库存选品弹窗

`ProductSelectModal` 用于入库、出库、调拨和盘点的选品。调整如下：

- 改用分页 `listProducts()` 请求，而不是 `listAllProducts()` 后在浏览器中筛选。
- 固定传递 `status=active`，只展示在售商品。
- 提供两个独立输入框：商品关键词（名称或简称）和条形码。
- 分类与品牌作为服务端请求条件。
- 保持后端分页；切换搜索条件或页码不会丢失已选择的商品。
- 库存选品弹窗不增加销售状态、成本价范围或售价范围控件。

## 接口契约

扩展现有 `GET /api/v1/products`，不新增独立高级搜索接口。

| 参数 | 含义 | 匹配方式 |
| --- | --- | --- |
| `keyword` | 商品名称或简称 | `name` 或 `short_name` 的模糊匹配 |
| `barcode` | 条形码 | 模糊匹配 |
| `category_id` | 分类 | 精确匹配 |
| `brand_id` | 品牌 | 精确匹配 |
| `status` | 销售状态 | 精确匹配 |
| `min_cost_price` | 最低成本价 | `cost_price >= value` |
| `max_cost_price` | 最高成本价 | `cost_price <= value` |
| `min_standard_price` | 最低售价 | `standard_price >= value` |
| `max_standard_price` | 最高售价 | `standard_price <= value` |

保留已有 `page` 和 `page_size` 分页参数。现有仅传名称、条形码、分类或状态的调用保持可用；前端商品列表改为传递 `keyword`，而不是将商品名称条件映射到 `name`。

## 实现边界

- API 路由仅解析查询参数并调用服务层。
- 商品查询、名称/简称的 OR 条件、品牌和价格范围条件都放在 `product_service.list_products`。
- 前端服务层的 `listProducts` 参数类型必须与后端查询参数一致。
- 商品列表通过现有 `ProTable` 搜索表单传参，不引入单独的搜索接口或客户端价格过滤。
- 选品弹窗在组件内缓存已选商品；库存操作页向弹窗提供当前行的基础商品信息，保证重新打开、翻页或筛选时仍可展示和取消已选项。

## 测试与验证

实施遵循测试优先：

1. 先补后端测试，覆盖名称命中、简称命中、品牌、成本价和售价上下限，以及区间包含边界。
2. 先补前端服务层测试，断言新增查询参数传递到 `GET /api/v1/products`。
3. 先补选品弹窗的纯逻辑测试，覆盖名称/简称与条形码的独立筛选条件，以及跨页或切换条件时保留已选商品。
4. 每个新增测试先验证为预期失败，再添加最小实现使其通过。
5. 完成后执行后端语法检查和测试，以及前端相关 Vitest、类型检查与 Biome lint。

## 非目标

- 不增加新的商品搜索接口。
- 不修改商品创建、编辑、价格变更或库存流水规则。
- 不为库存选品弹窗增加价格范围或销售状态筛选控件。
