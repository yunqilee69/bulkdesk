# 商品价格表单调整设计

## 背景

当前商品价格相关交互存在两个问题：

1. 新建商品以及价格管理中的标准售价、成本价保存，都强制要求填写调价原因。
2. 新建商品时无法同时设置各会员等级的会员价。

## 目标行为

### 原因字段

- 新建商品保留“初始定价原因”，但改为选填。
- 价格管理保留“调整原因”，但标准售价、成本价和会员价保存时均不强制填写。
- 未填写原因时，价格变更日志使用空字符串记录原因。

### 新建商品

- 新建商品弹窗继续要求填写标准售价和成本价。
- 新建商品弹窗新增“会员价格”区域，展示当前全部会员等级。
- 每个会员等级的会员价均为选填，可只设置部分等级。
- 商品基础信息、标准售价、成本价和已填写的会员价通过一次创建请求提交。
- 后端在同一数据库事务中创建商品、会员价和对应价格变更日志，避免部分成功。

### 编辑商品

- 编辑商品弹窗只维护商品基础资料。
- 编辑时不展示标准售价、成本价、初始定价原因或会员价。
- 商品创建后的全部价格调整继续统一通过“价格管理”弹窗完成。

## 接口契约

`POST /api/v1/products` 的请求体增加可选字段：

```json
{
  "member_prices": [
    {
      "level_id": "会员等级 ID",
      "price": 88.0
    }
  ]
}
```

- `price_reason` 改为可选，缺省值为空字符串。
- `member_prices` 改为可选，缺省值为空数组。
- 同一个会员等级不可重复提交。
- 所有会员等级必须真实存在，否则整个创建请求失败。
- `PriceChangeRequest.reason` 改为可选，缺省值为空字符串。

## 前端数据流

1. 商品页面初始化时并行加载分类、品牌和会员等级。
2. 打开新建弹窗时，根据会员等级生成空白会员价输入行。
3. 提交时仅收集已填写价格的会员等级，组成 `member_prices`。
4. 空白原因不发送或发送空字符串，后端均按空字符串处理。
5. 打开编辑弹窗时不生成或展示会员价输入区域。

## 验证范围

- 后端 schema 允许省略初始定价原因和调价原因。
- 后端创建商品时保存已填写的会员价及价格日志。
- 后端拒绝重复或不存在的会员等级，并且不写入商品。
- 前端只提交已填写的会员价。
- 新建表单展示会员价，编辑表单不展示任何价格字段。
- 现有价格管理的标准售价、成本价、会员价保存均允许空原因。

## 非目标

- 不修改数据库表结构。
- 不在编辑商品弹窗中维护价格。
- 不改变现有价格管理弹窗的价格趋势图和会员价批量保存方式。
